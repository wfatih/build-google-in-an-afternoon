# Production Deployment Recommendations

## Scaling the Crawler

The most impactful change for production would be replacing the single-process model with a distributed work queue (e.g., Redis Streams, Kafka, or AWS SQS) so that many crawler workers can run across separate machines. Each worker would pull a `(url, origin, depth)` item, fetch and parse the page, write the result to a shared index store, and push child URLs back onto the queue. The visited-URL set — currently an in-memory Python `set` — would become a Redis Set or a Bloom filter (to trade a small false-positive rate for dramatically reduced memory and network overhead at billions-of-URL scale). Rate limiting would move from the per-process token bucket to a distributed rate limiter (e.g., a Redis-backed sliding window per target domain), ensuring polite crawling even when dozens of workers share the same destination host. `robots.txt` compliance, retry logic with exponential back-off, and per-domain politeness delays would also be added at this layer.

## Scaling the Index and Search

The flat JSON inverted index would be replaced by a purpose-built search engine such as Elasticsearch or OpenSearch, which provides distributed sharding, replication, and a rich query DSL. For the indexing pipeline specifically, pages would be written to an event stream as they are crawled; a separate indexing service would consume that stream and bulk-import into the search backend, decoupling crawl throughput from index write latency. Relevancy would be improved by adding TF-IDF or BM25 scoring, title-text boosting, anchor-text signals, and eventually a lightweight PageRank computed periodically over the crawled link graph. On the operational side, both crawler workers and the search API would be containerised (Docker/Kubernetes) with horizontal auto-scaling, health checks, circuit breakers against slow remote hosts, and structured logging fed into an observability stack (e.g., Prometheus + Grafana) tracking key SLOs: crawl throughput (pages/hour), search p99 latency, and index freshness lag.
